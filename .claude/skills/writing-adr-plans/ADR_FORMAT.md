# ADR フォーマットリファレンス

## 原則

- **簡潔さを最優先**。散文より比較表と箇条書きで伝える
- Context は問題の核心と「なぜ今この判断が必要か」が端的に伝わるように
- Decision は比較表 + 根拠の箇条書き。長い説明文は避ける
- Consequences は 3〜5 項目程度

## 構造テンプレート

```markdown
# [タイトル]

Date: YYYY-MM-DD
Status: Accepted

## Context

[問題・状況・制約の説明。なぜこの判断が必要になったかを記述]

## Decision

### [決定の見出し]

| 観点  | 選択肢A  | 選択肢B | 選択肢C |
| ----- | -------- | ------- | ------- |
| 基準1 | 値       | 値      | 値      |
| 基準2 | **強調** | 値      | 値      |

- [根拠1。外部リンクで裏付け](https://...)
- [根拠2]
- [根拠3]

## Consequences

- [肯定的な結果]
- [否定的な結果・トレードオフ]
- [将来の考慮事項]
```

## 必須ルール

- **セクション順は固定**: Context → Decision → Consequences
- **比較表**: 選択肢が複数ある場合は必ずテーブル。最低3列（観点 + 選択肢）
- **根拠の裏付け**: GitHub リンク、公式ドキュメント、コミュニティ採用実績など具体的証拠
- **相互参照**: 関連 ADR を `[ADR: タイトル](./YYYY-MM-DD-slug.md)` でリンク

## 比較表のパターン

観点列は文脈に応じて使い分け:

```markdown
| 観点 | A | B | ← ツール比較
| 項目 | 元の決定 | 変更後 | ← Addendum の変更表
```

表中の強調: `**太字**` で推奨・優位を示す。`✅` `❌` `⚠️` はオプション。

## Decision サブセクション

選択肢を詳細に比較する場合:

```markdown
### A. [選択肢名]

- ✅ [利点]
- ❌ [欠点]

### B. [選択肢名]

- ✅ [利点]
- ❌ [欠点]
```

## Addendum（後から決定を更新する場合）

```markdown
## Addendum (YYYY-MM-DD): [変更の要約]

Status: **Partially Superseded** — [何が変わったか]

### 経緯

[変更に至った理由]

### 変更した決定

| 項目 | 元の決定 | 変更後 |
| ---- | -------- | ------ |
| ...  | ...      | ...    |
```

## Consequences の書き方

肯定的・否定的・中立をバランスよく含める:

```markdown
- `nix develop` で自動的に pre-commit hooks が有効化される。セットアップ手順ゼロ
- flake-parts の module system を理解する必要があるが、基本的な perSystem / flake の2分割は直感的
- 将来 Base16 パレットマッピングが改善された場合、再導入は flake input 追加で対応可能
```
