# workspace CLI の技術選定と設計方針

Date: 2026-04-08
Status: Accepted

## Context

tmux のセッション管理に以下の課題がある:

- 既存の `tm` コマンドが 1command 1action で、一覧表示中に削除やリネームに移行できない
- Claude Code 等の AI エージェントの状態（入力待ち・完了）が macOS 通知センターに埋もれて見逃す
- リポジトリごとのレイアウト（タブ・ペイン・起動コマンド）を毎回手動で構築している

これらを解決する `workspace` コマンドの技術選定と設計方針を決定する。

## Decision

### 実装言語・フレームワーク

| 観点           | zsh + fzf                | Go + bubbletea |
| -------------- | ------------------------ | -------------- |
| 起動速度       | **即座**                 | 数十ms         |
| 依存           | **なし（既存環境）**     | Go ランタイム  |
| tmux 連携      | **直接コマンド呼び出し** | exec 経由      |
| ツリー表示     | △ 疑似的                 | **ネイティブ** |
| シームレス操作 | △ fzf --bind で部分的    | **完全**       |
| 開発コスト     | **低い**                 | 高い           |

- 要件の大半（一覧表示、フィルタ、複数アクション）は fzf の `--bind` + `reload` で実現可能
- AI 状態表示・通知バッジは fzf の行にテキストラベルを付与するだけで実現可能

**決定**: zsh + fzf で CLI として実装する。

### レイアウトテンプレート

| 観点                | 自作 | smug                     | tmuxinator        | tmuxp         |
| ------------------- | ---- | ------------------------ | ----------------- | ------------- |
| 言語                | -    | **Go**                   | Ruby              | Python        |
| 定義方式            | -    | **宣言的 YAML**          | 宣言的 YAML       | YAML/JSON     |
| ペイン単位コマンド  | -    | **○**                    | ○                 | ○             |
| ローカル設定        | -    | **`.smug.yml` 自動検知** | `.tmuxinator.yml` | `.tmuxp.yaml` |
| ライフサイクル hook | -    | **before_start / stop**  | 限定的            | なし          |
| Nix パッケージ      | -    | **あり**                 | あり              | あり          |
| Stars               | -    | **877**                  | 13.5k             | 4.5k          |

- [smug](https://github.com/ivaaaan/smug) は Go 製でバイナリ依存が軽く、Nix との相性が良い
- `.smug.yml` をリポジトリ直下に置けば自動検知。`before_start` / `stop` hook で Docker 等のライフサイクル管理も可能
- `manual: true` で特定ウィンドウを `-w` 指定時のみ起動できる柔軟性がある

**決定**: smug を採用する。

### AI エージェント状態の取得方式

| 観点           | Hooks でファイル書き出し | tmux ペインスキャン | プロセス検査 |
| -------------- | ------------------------ | ------------------- | ------------ |
| 信頼性         | **◎**                    | ○                   | ×            |
| 導入コスト     | **低（既存 hook 拡張）** | 低                  | -            |
| UI 変更耐性    | **◎**                    | ×                   | -            |
| セッション特定 | **○**                    | ○                   | ×            |

- Claude Code の hooks（`Stop`, `Notification`, `PreToolUse` 等）で状態を `$TMPDIR` 配下のファイルに書き出す
- 既存の notification hook を拡張するだけで導入可能
- ペインスキャンは Claude Code の UI 変更に弱いため、hooks を主軸にする

**決定**: Claude Code hooks でセッションごとの状態ファイルに書き出す方式を採用する。

### 通知ロジックの集約

- 通知の送信・既読・バッジ出力のロジックを `workspace-notify` スクリプトに集約する
- Claude Code hooks、tmux.conf、fzf からは `workspace-notify` のインターフェースを呼ぶだけ
- ファイルパスや形式の知識は `workspace-notify` 内部に閉じる

**決定**: 通知ロジックは `workspace-notify` に集約し、呼び出し側にはトリガー設定（1行）のみ配置する。

## Consequences

- fzf ベースのため、ツリー表示（折りたたみ/展開）やペインレベルのジャンプは実現できない
- smug は外部依存だが、Nix パッケージがあるため管理コストは低い
- AI 状態ファイルは `$TMPDIR` に置くため再起動で消える。永続化が必要になった場合は保存先の変更が必要
- 通知トリガーの設定が Claude Code settings.json と tmux.conf に散在するが、各ファイルには `workspace-notify` を呼ぶ1行のみで、ロジックは集約されている
